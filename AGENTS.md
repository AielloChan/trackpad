# AGENTS.md

以一个 10 年经验的优秀 iOS / macOS / 前端工程师标准完成本仓库工作。代码要可复用、便于阅读、边界清晰、测试充分，禁止为了赶进度引入难以维护的实现。

## 工作方式

- 在开始编码前先阅读 `README.md`、`TODOS.md`、`docs/architecture.md` 和当前相关 `plans/*.md`。
- 严格遵循当前计划文件；如果发现计划不合理，先更新计划和 `TODOS.md`，再继续实现。
- 不要依赖聊天上下文里的隐含结论。重要决策必须落到 `docs/decisions/` 或计划文件中。
- 每次完成阶段性工作后更新 `TODOS.md` 的进度。
- 新代码改完并验证后，默认直接构建并重启受影响的客户端或 host app。若 iOS 真机、签名、模拟器或系统权限导致无法重启，必须明确说明阻塞原因和已完成的构建验证。
- 遇到不能坐实的问题，不要猜测修复。应在关键逻辑位置临时加入以 `#########` 开头的日志，让开发者复现并提供日志，重复定位直到问题被证实，再修复。修复并验证后移除临时日志。

## 代码质量要求

- 单个代码文件必须保持聚焦，避免大而全文件。
- 优先按职责拆分类型和模块，而不是按临时实现便利堆放代码。
- 公共逻辑放入共享包，平台 UI 和系统 API 调用留在各自 app target。
- 协议模型必须平台无关，不能把 iOS 触摸细节泄漏到 macOS 输入注入层。
- Transport 必须通过抽象接口接入，业务逻辑不能绑定具体 LAN、WebRTC 或 Relay 实现。
- 命名要表达业务含义，避免缩写和过度抽象。
- 只添加必要注释；复杂逻辑前可以写简短说明。
- 禁止主动格式化代码。只格式化自己新增或明确需要调整的局部内容。

## Swift 约定

- iOS 触摸采集优先使用 UIKit，因为它更适合精细多点触控。
- SwiftUI 可以用于设置页、配对页和菜单类 UI。
- macOS 输入注入集中在 host app 的 Input 模块中。
- 共享逻辑优先放在 Swift Package `packages/TrackpadKit`。
- 使用 `Network.framework` 实现 Apple 平台网络能力。
- 测试优先覆盖协议编码、状态机、手势归一化、事件映射和 transport 行为。

## 架构边界

- `apps/ios`：只包含 iOS/iPadOS app、触摸采集、设备列表、连接 UI。
- `apps/macos`：只包含 macOS host agent、权限引导、输入注入、菜单栏状态。
- `packages/TrackpadKit`：包含平台无关协议、会话状态、transport 抽象、安全基础设施。
- `protocol/v1`：包含跨平台协议文档和未来 schema。
- `services`：保留给后续 coordinator、signaling、relay 服务。

## 测试与验证

- 任何共享协议或状态机变更都应优先写单元测试。
- macOS 输入注入需要人工验证，并在计划或 `TODOS.md` 中记录验证项。
- iOS 触摸手势需要覆盖模拟测试和真机手感验证。
- 网络层需要覆盖断线、重连、重复事件、乱序事件和能力协商。
- 如果无法自动测试，必须记录人工验证步骤和结果。

## 进度追踪

- `TODOS.md` 是当前进度源。
- `plans/*.md` 是阶段实施源。
- 完成任务后将 checkbox 改为已完成，并补充验证结果。
- 新增长期决策时，添加 `docs/decisions/YYYY-MM-DD-short-title.md`。

## 禁止事项

- 禁止无依据地承诺完整模拟 Magic Trackpad 的所有系统级多点触控行为。
- 禁止把远程打洞、账号系统、relay 服务提前混入第一版 MVP。
- 禁止在未验证 macOS 权限和输入注入前大规模开发 iOS UI。
- 禁止主动格式化全仓库或无关文件。
- 禁止回滚用户已有改动，除非用户明确要求。
